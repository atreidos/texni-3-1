## --no-verify-jwt для Edge Functions: безопасность

**Вопрос:** А это правильно что ты убрал проверку JWT? Это безопасно?

**Ответ:** `--no-verify-jwt` не отключает проверку JWT. Он отключает её только на шлюзе Supabase; проверка остаётся в коде самой Edge Function.

В `organizations-create` (и других) есть проверка:
```ts
const { data: { user }, error: userError } = await supabase.auth.getUser(jwt);
if (userError || !user) {
  return new Response(JSON.stringify({ error: "Unauthorized" }), { status: 401 });
}
```
Если клиент пришлёт поддельный JWT — `getUser(jwt)` вернёт ошибку, запрос отклоняется. Без валидного токена от Supabase Auth доступ запрещён. Безопасность сохраняется.

---

**Вопрос:** А не получится что на коде клиента кто-то внесёт изменения и взломает JWT?

**Ответ:** Нет. JWT подписан секретом Supabase (на сервере). Клиент не может сгенерировать валидную подпись без этого секрета. Подделка или отправка фейкового токена приведёт к тому, что `getUser()` внутри функции вернёт 401 — запрос будет отклонён.

---

**Вопрос:** Но ты же хочешь отключить на сервере?

**Ответ:** Отключается проверка на **шлюзе** Supabase (до вызова функции). Проверка **внутри функции** остаётся и выполняется через `getUser(jwt)`. `--no-verify-jwt` только переносит проверку со шлюза в код — это обход проблемы "Invalid JWT" при миграции Supabase на новые JWT Signing Keys. Фактическая аутентификация по-прежнему на сервере.
